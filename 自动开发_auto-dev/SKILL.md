# 自动开发 (Auto Dev) - 往死里干模式

> **角色**：自主开发执行者（Tech Lead）
> **目标**：需求文档 → 可交付成品，全程无人值守
> **原理**：多人协作 + 严格 TDD + 完成前验证
> **场景**：白天写需求，晚上跑，早上交付
> **效率**：相比串行执行，提升 **2-5 倍**

## 核心铁律

| 铁律 | 说明 |
|------|------|
| **AC 单一来源** | 验收标准只在 /clarify 定义，后续只能引用不能重新定义 |
| **测试先行** | /plan 之后、开发之前必须执行 /test-gen 生成 FAILING 测试 |
| **严格 TDD** | 没有先失败的测试，就没有生产代码 |
| **完成前验证** | 只有亲眼看到验证命令成功，才能声称完成 |
| **服务启动验证** | 必须亲眼看到服务正常启动并响应请求 |
| **文件边界** | 多人协作时，每个开发者只修改分配的文件 |

## 前置条件（门控检查）

### 1. AC 文档（必须）

> ⚠️ **门控铁律**：必须先执行 `/clarify` 生成 AC 文档，这是验收标准的唯一来源

**门控检查**：
```bash
# 检查 AC 文档是否存在
CLARIFY_DOC=$(ls docs/需求文档/clarify_*.md 2>/dev/null | head -1)
if [ -z "$CLARIFY_DOC" ]; then
  echo "❌ 门控失败: AC 文档不存在"
  echo "   修复: 先执行 /clarify 生成需求文档和 AC 表格"
  exit 1
fi
echo "✅ AC 文档: $CLARIFY_DOC"
```

**AC 文档必须包含**：
- 验收标准（AC 表格，Given-When-Then 格式）
- 正常流程 + 异常流程 + 边界情况

**如果没有 AC 文档**：
1. 先执行 `/clarify` 生成 AC 文档
2. 然后再执行 `/auto-dev`

### 2. 项目环境

- 项目已初始化（git、依赖配置）
- CLAUDE.md 存在（技术栈、规范说明）
- 测试框架已配置（pytest/vitest 等）

## 执行流程

### 阶段 0：门控检查（必须通过）

```
┌────────────────────────────────────────────────────────────────┐
│ 🔒 门控: AC 文档存在                                            │
│    检查: docs/需求文档/clarify_[功能名].md 是否存在              │
│    失败: 先执行 /clarify 生成需求文档和 AC 表格                   │
└────────────────────────────────────────────────────────────────┘
```

**门控失败时**：停止执行，提示用户先执行 `/clarify`

### 阶段 1：设计

1. **读取 AC 文档**：`docs/需求文档/clarify_[功能名].md`
2. **执行 /explore**：调研最佳实践（至少 5 个来源）
3. **执行 /design**：输出架构设计
4. **执行 /critique**：批评者视角审查架构设计，找出遗漏和风险
5. **执行 /plan**：输出多人协作实施计划
   - 识别可并行的独立功能
   - 分配开发者（Alice/Bob/Charlie）
   - 指定文件范围（避免冲突）
   - **引用 /clarify 的 AC**（禁止重新定义）
6. **执行 /critique**：批评者视角审查计划
7. **保存设计文档到 docs/ 目录**

### 阶段 2：测试先行

> ⚠️ **铁律**：没有测试就没有开发，测试是开发的验收标准

1. **执行 /test-gen from-clarify**：从 AC 文档生成 FAILING 测试
2. **验证测试文件存在**：`tests/test_[功能名]_acceptance.py`
3. **运行测试确认失败**：符合 TDD 预期

**如果跳过此阶段**：后续开发将无法通过 `/check` 的 TDD 验证

### 阶段 3：Tech Lead 前置工作

1. **搭建基础设施**
   - 数据库 Schema
   - 中间件/拦截器
   - 路由配置
   - 前端 Context/Store
2. **验证基础设施**
   - 数据库连接正常
   - 服务能启动（不报错）

### 阶段 4：开发者并行执行（核心提效）

> **遵循 /run-plan 的 TDD 流程**：基于已有的 FAILING 测试进行开发

**同时启动多个子代理（使用 Task 工具并行）**：

```
┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│   Alice     │  │    Bob      │  │   Charlie   │
│  登录功能   │  │  注册功能   │  │  密码重置   │
│ (前+后端)   │  │ (前+后端)   │  │ (前+后端)   │
│ 严格 TDD    │  │ 严格 TDD    │  │ 严格 TDD    │
└─────────────┘  └─────────────┘  └─────────────┘
```

**每个开发者必须**：
- 🔴 **基于已有测试**：测试已由 /test-gen 生成，开发者只需让测试通过
- 🔴 **严格 TDD**：运行测试 → 看到失败 → 写最小实现 → 看到通过（见 `TDD规范.md`）
- ✅ **完成前验证**：亲眼看到验证命令成功（见 `完成前验证.md`）
- 🚧 **文件边界**：只修改分配的文件，不越界

### 阶段 5：Tech Lead 后置工作（铁律）

1. **代码合并与冲突检测**
2. **集成测试**
3. **服务启动验证**（铁律）
   ```bash
   # 必须亲眼看到服务正常启动并响应请求
   python -m uvicorn app.main:app &
   sleep 3
   curl -f http://localhost:8000/health  # 必须看到 200 OK
   ```
4. **E2E 测试**（如适用）

### 阶段 6：检查（9 Agent 并行）

1. **执行 /check**（v4.0）：
   - Agent1: 铁律检查
   - Agent2: 后端自动化
   - Agent3: 前端自动化
   - Agent4: 代码质量审查
   - Agent5: 文档同步检查
   - Agent6: 服务启动验证
   - Agent7: 需求覆盖检查
   - **Agent8: TDD 流程验证**
   - Agent9: Hooks 配置检测

### 阶段 7：测试验收

1. **执行 /qa**：金字塔门控测试（单元→集成→E2E）
2. **核对验收标准**：基于 /clarify 的 AC 表格逐项确认

### 阶段 8：修复（如有问题）

1. **分析失败原因**（使用 /debug）
2. **修复问题**（严格 TDD，不能跳过）
3. **完成前验证**（必须亲眼看到成功）
4. **重新执行 /check**（回到阶段 6）
5. **循环直到通过**

### 阶段 9：完成

以下条件**全部满足**时才算完成：

- [ ] 所有开发者的 TDD 执行记录完整（每个测试都先看到失败）
- [ ] 所有开发者的完成前验证都通过
- [ ] **服务启动验证通过**（亲眼看到服务正常运行）
- [ ] /check 通过（6 Agent 全绿）
- [ ] /qa 验收通过
- [ ] 验收标准全部满足
- [ ] 输出完成标记：`<promise>COMPLETE</promise>`

### AC 流转链路（铁律）

```
/clarify 定义 AC（单一来源）
    ↓
/plan 引用 AC（禁止重新定义）
    ↓
/test-gen 从 AC 生成测试
    ↓

```

**注意**：
- `/clarify` **必须**在 `/auto-dev` 之前执行（AC 是验收的唯一来源）
- `/ship` 建议手动执行（最终交付需人工确认）

## 适用场景

### ✅ 适合使用

| 场景 | 说明 |
|------|------|
| 需求清晰的新功能 | 有完整 PRD，验收标准明确 |
| 独立模块开发 | 与现有代码耦合度低 |
| CRUD 类功能 | API、后台管理页面 |
| 过夜运行 | 白天写需求，晚上跑 |

### ❌ 不适合使用

| 场景 | 原因 | 替代方案 |
|------|------|---------|
| 需求模糊 | 无法自动澄清 | 先 /clarify |
| 复杂重构 | 需要人工判断 | /refactor |
| 紧急 Bug | 需要快速响应 | /debug |
| 核心架构改动 | 风险高 | 手动流程 |

## 最佳实践

### 1. 需求文档要详细

❌ **不好**：
```
实现用户管理功能
```

✅ **好**：
```markdown
# 用户管理功能

## 功能要求
1. 用户列表：分页、搜索、排序
2. 创建用户：表单验证、唯一性检查
3. 编辑用户：部分更新
4. 删除用户：软删除、确认提示

## 技术约束
- 后端：FastAPI + SQLAlchemy
- 前端：React + Ant Design
- 认证：JWT

## 验收标准
- [ ] 所有 API 端点实现（GET/POST/PUT/DELETE）
- [ ] 前端页面完成（列表、表单、详情）
- [ ] 单元测试覆盖率 > 80%
- [ ] E2E 测试通过
```

### 2. 设置合理的迭代上限

| 功能规模 | 建议迭代 | 预估成本 |
|---------|---------|---------|
| 小功能（1-3 文件） | 10-15 次 | $10-30 |
| 中等功能（5-10 文件） | 20-30 次 | $30-80 |
| 大功能（10+ 文件） | 40-60 次 | $80-150 |

### 3. 分阶段执行

对于大型功能，建议拆分：

```bash
# 阶段 1：后端 API
/auto-dev docs/PRD_用户管理_后端.md --max-iterations 30

# 阶段 2：前端页面
/auto-dev docs/PRD_用户管理_前端.md --max-iterations 30
```

## 故障处理

### 1. 迭代用完但未完成

```bash
# 检查当前状态
cat .claude/auto-dev-detail.md | tail -50

# 增加迭代继续（Claude 会自动识别已完成的工作）
~/.claude/skills/自动开发_auto-dev/scripts/auto-dev.sh \
  --prd docs/需求文档/PRD_xxx.md \
  --max-iterations 20
```

### 2. 测试一直失败

```bash
# 手动介入调试
/debug

# 修复后重新运行（Claude 会识别 git 历史，不会重复已完成的工作）
/auto-dev docs/需求文档/PRD_xxx.md
```

### 3. 卡在某个步骤

```bash
# 查看日志定位问题
cat .claude/auto-dev.log | tail -100

# 手动执行该步骤
/check  # 或其他卡住的步骤

# 重新运行自动开发
/auto-dev docs/需求文档/PRD_xxx.md
```
