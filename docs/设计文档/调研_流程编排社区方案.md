# 调研：Claude Code 多步骤研发流程自动化编排方案

> 调研日期：2026-02-13
> 调研目标：系统梳理业界在 Claude Code 多步骤研发流程编排方面的已有方案，为实现 design -> plan -> run-plan -> check -> qa -> fix(循环) -> ship 自动化编排提供决策依据。

---

## 一、核心需求分析

用户需求的关键特征：
1. **多步骤顺序执行**：design -> plan -> run-plan -> check -> qa -> fix -> ship
2. **独立上下文**：每个环节独立 context window，避免上下文污染
3. **文档留痕串联**：步骤间通过文档传递状态，可追溯
4. **循环能力**：qa -> fix 可循环直到通过
5. **自动化**：需求澄清后自动执行，减少人工干预

## 二、GitHub 项目/工具

### 2.1 obra/superpowers（42k+ stars，活跃维护）

- **链接**：https://github.com/obra/superpowers
- **核心思路**：一套完整的 agentic skills 框架，实现 7 阶段工作流：需求澄清 -> spec 确认 -> 实现计划 -> subagent 逐任务实现 -> code review -> 完成。skills 自动激活（如 test-driven-development 在实现时自动加载）。
- **与需求的匹配度**：
  - 有完整的 spec -> plan -> implement -> review 流程
  - 使用 subagent-driven-development 实现两阶段 review
  - 自动创建 git worktree 隔离并行任务
  - 有 remembering-conversations skill 实现跨 session 记忆
- **优点**：
  - 社区最成熟、star 数最高的方案
  - 官方 Anthropic marketplace 收录
  - 通过 plugin 机制安装，开箱即用
  - 方法论驱动（TDD、YAGNI、DRY）
- **缺点**：
  - 不是严格的"独立上下文 + 文档留痕"模式，更偏向 session 内 subagent 协作
  - 流程编排不够显式，依赖 skill 自动激活
  - 无法自定义步骤顺序和循环逻辑
- **适用场景**：成熟的日常开发工作流，但不直接解决"多步骤独立上下文编排"

### 2.2 ruvnet/claude-flow（14k+ stars，活跃维护）

- **链接**：https://github.com/ruvnet/claude-flow
- **核心思路**：MCP 协议驱动的多 agent 编排平台。支持 hierarchical（queen/workers）和 mesh（peer-to-peer）模式。通过 MCP server 提供 87+ 工具，支持 parallel/sequential/adaptive/balanced 策略。
- **与需求的匹配度**：
  - 支持 Pipeline 模式（sequential chaining）
  - 支持 agent 依赖管理和 DAG 任务图
  - 有 SPARC 方法论（Specification -> Pseudocode -> Architecture -> Refinement -> Completion）
  - 支持 stream-json chaining 实现 agent 间实时通信
- **优点**：
  - 功能最全面的第三方编排框架
  - 5 种执行模式（Autopilot、Ultrapilot、Swarm、Pipeline、Ecomode）
  - 自学习能力，成功模式会被记忆复用
  - 支持多 LLM（Claude/GPT/Gemini/本地模型）
- **缺点**：
  - 项目体量大，引入成本高
  - 部分功能过度工程化
  - 社区反馈：与 Claude Code 原生 Team 功能有概念重叠
  - "Ranked #1" 的自我营销较重
- **适用场景**：大型企业级多 agent 编排，功能需求复杂

### 2.3 wshobson/agents（17.6k stars，活跃维护）

- **链接**：https://github.com/wshobson/agents
- **核心思路**：112 个专用 agent + 16 个工作流编排器 + 73 个 plugin。提供 Context -> Spec & Plan -> Implement 结构化工作流，带 TDD 验证检查点。
- **与需求的匹配度**：
  - 16 个编排器覆盖 full-stack、security、ML pipeline 等场景
  - 支持 Agent Teams（parallel code review / hypothesis debugging / coordinated feature dev）
  - 有 TDD workflow 验证检查点
  - 支持 session 间状态持久化
- **优点**：
  - 按功能模块化安装（73 个独立 plugin），token 效率高
  - 编排器种类丰富，覆盖面广
  - 7 agent 顺序链：backend-architect -> database-architect -> frontend-dev -> test-automator -> security-auditor -> deployment-engineer -> observability-engineer
- **缺点**：
  - 插件数量过多，选择成本高
  - 重量级方案，需要大量配置
  - 与需求的"独立上下文 + 文档留痕"模式不完全匹配
- **适用场景**：企业级全栈开发团队

### 2.4 catlog22/Claude-Code-Workflow

- **链接**：https://github.com/catlog22/Claude-Code-Workflow
- **核心思路**：JSON 驱动的多 agent 开发框架，4 级工作流系统。支持 brainstorm:auto-parallel（多角色并行分析）和 plan -> execute 顺序执行。
- **与需求的匹配度**：
  - 4 级工作流（lite -> plan -> TDD plan -> brainstorm+plan+execute）
  - 每个 task 是 `.json` 文件，天然文档留痕
  - 有 issue workflow：discover -> plan -> queue -> execute（worktree）
  - 支持 npm 安装（`npm install -g claude-code-workflow`）
- **优点**：
  - JSON 驱动，任务定义清晰可追溯
  - 支持 npm 全局安装，跨项目复用
  - 4 级工作流可按复杂度选择
  - 支持 worktree 隔离
- **缺点**：
  - 社区较小，star 数待确认
  - 多 CLI 编排（Gemini/Qwen/Codex）增加复杂度
  - 中文文档可用
- **适用场景**：中等复杂度项目，偏好 JSON 配置

### 2.5 parcadei/Continuous-Claude-v3

- **链接**：https://github.com/parcadei/Continuous-Claude-v3
- **核心思路**：通过 hooks 维护 ledger（账本）和 handoff（交接文档），实现跨 session 上下文持续。"Compound, don't compact"——每次 session 结束自动提取学习成果，下次 session 启动时加载。
- **与需求的匹配度**：
  - **直接解决"独立上下文 + 文档留痕"问题**
  - hooks 自动生成 handoff 文档（包含最近改动、学习成果、下一步计划）
  - thoughts/ledgers/ 目录存放持续性账本
  - thoughts/shared/handoffs/ 存放 session 间交接文档（YAML）
  - thoughts/shared/plans/ 存放实现计划
  - "maestro" agent 自动编排不同类型的 agent
- **优点**：
  - 最接近"独立上下文 + 文档留痕"需求的方案
  - hooks 驱动，自动化程度高
  - 有实时 StatusLine 显示上下文使用率
  - 5 层代码分析系统 + 语义索引
- **缺点**：
  - 项目复杂度高（32 agents + 30 hooks + 109 skills）
  - 需要 PostgreSQL + pgvector
  - 学习曲线陡峭
- **适用场景**：需要跨 session 持续性和文档留痕的复杂项目

### 2.6 shinpr/claude-code-workflows

### 2.7 frankbria/ralph-claude-code

  - 无法定义步骤间的文档传递
- **适用场景**：单一大任务的自主迭代完成

## 三、Claude Code 官方方案

### 3.1 Custom Subagents（正式功能）

- **文档**：https://code.claude.com/docs/en/sub-agents
- **核心能力**：
  - 每个 subagent 独立 context window + 自定义 system prompt + 工具权限
  - 支持 `.claude/agents/` 目录下 Markdown + YAML frontmatter 定义
  - 可通过 Agent SDK 的 `agents` 参数编程定义
  - 支持 model 选择（Opus/Sonnet/Haiku/inherit）
  - 支持 skills 注入
- **限制**：
  - subagent 不能嵌套 spawn subagent
  - 工作在单个 session 内，不是独立进程
  - 有 turn budget 限制
- **适用**：session 内的轻量级任务委派

### 3.2 Agent Teams（实验性功能）

- **文档**：https://code.claude.com/docs/en/agent-teams
- **启用方式**：设置 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS`
- **核心能力**：
  - team lead 协调多个 Claude Code 实例
  - 每个 teammate 独立 context window，可直接互相通信
  - TaskCreate 支持 DAG 依赖（Task 3 等待 Task 1 + Task 2）
  - 共享任务列表（通过 `CLAUDE_CODE_TASK_LIST_ID` 环境变量）
  - 状态持久化在 `~/.claude/teams/` 和 `~/.claude/tasks/`
- **与需求的匹配度**：
  - **独立上下文**：每个 teammate 独立 context window
  - **DAG 依赖**：支持任务间依赖关系
  - **通信**：teammate 可互相发消息共享发现
- **限制**：
  - 实验性功能，API 可能变化
  - 团队协作模式，不完全匹配"顺序流水线"
  - 文档留痕需要自行实现

### 3.3 Headless Mode（claude -p）

- **文档**：https://code.claude.com/docs/en/headless
- **核心能力**：
  - `claude -p "prompt"` 非交互执行
  - `--output-format json/stream-json` 程序化解析
  - `--resume session_id` 跨步骤延续 session
  - `--max-turns` 限制 agent 循环
  - `--permission-mode plan` 只读探索模式
- **适用**：bash 脚本编排的基础能力

### 3.4 Hooks 系统

- **文档**：https://code.claude.com/docs/en/hooks-guide
- **核心能力**：
  - 10 种事件：PreToolUse、PostToolUse、UserPromptSubmit、SessionStart/End、SubagentStop 等
  - PreToolUse 可修改工具输入、block/allow/ask
  - PostToolUse 可触发自动格式化、测试、git 操作
  - Stop hook 可在 Claude 完成时触发下一步
  - Hook timeout 10 分钟
- **适用**：步骤间自动触发的轻量编排

### 3.5 Plugins / Skills 系统

- **核心能力**：
  - Skills 文件（SKILL.md）支持自动激活（model-invoked）和手动激活（user-invoked）
  - 支持 `disable-model-invocation` 和 `user-invocable` 控制
  - description 预算：context window 的 2%
  - Plugins 支持从 marketplace 安装
- **适用**：定义每个流程步骤的执行逻辑

### 3.6 Agent SDK

- **文档**：https://platform.claude.com/docs/en/agent-sdk
- **核心能力**：
  - TypeScript/Python SDK 编程调用 Claude Code
  - 支持 `agents` 参数定义 subagent
  - 支持 `resume: sessionId` 跨调用延续
  - 结构化输出（tool_result）可作为 GitHub Action output
- **适用**：编程实现完全自定义的编排逻辑

## 四、社区实践模式

### 4.1 Bash 脚本 + claude -p 顺序执行（最常见模式）

```bash
# 伪代码示例
claude -p "设计方案" --output-format json > design.json
claude -p "基于 design.json 制定计划" --output-format json > plan.json
for task in $(jq -r '.tasks[]' plan.json); do
  claude -p "执行任务: $task" --dangerously-skip-permissions
done
claude -p "运行检查" --output-format json > check.json
# ... 循环修复
```

- **优点**：简单直接，每步独立上下文，文档留痕（JSON 输出）
- **缺点**：步骤间上下文完全丢失，需要精心设计 prompt 传递信息

### 4.2 SPARC 方法论 Shell 脚本（ruvnet/sparc）

- **链接**：https://github.com/ruvnet/sparc
- **流程**：Specification -> Pseudocode -> Architecture -> Refinement -> Completion
- **实现**：`claude-sparc.sh` 按阶段调用 claude -p
- **问题**：社区反馈"每个阶段从零开始无上下文"

### 4.3 Session Resume 链式执行

```bash
# 获取 session ID，后续步骤 resume 同一 session
result=$(claude -p "设计" --output-format json)
session_id=$(echo $result | jq -r '.session_id')
claude --resume "$session_id" -p "制定计划"
claude --resume "$session_id" -p "执行计划"
```

- **优点**：保持上下文连贯
- **缺点**：上下文会逐步膨胀，后期步骤质量下降（Issue #21776 描述的问题）

### 4.4 Handoff 文档模式（Continuous-Claude 创造）

- **核心思路**：每步结束时自动生成 handoff.yaml，下步启动时加载
- **handoff 内容**：最近改动（file:line）、学习成果、下一步计划
- **实现**：hooks（SessionEnd 写 handoff，SessionStart 读 handoff）

### 4.5 Orchestrator + Subagent 模式

- **核心思路**：主 agent 作为编排器，将每个步骤委派给专用 subagent
- **实现**：CLAUDE.md 中定义编排协议，subagent 文件定义每步的角色和工具
- **优点**：利用 Claude Code 原生能力
- **缺点**：受限于单 session context window

### 4.6 自检查点重置模式（Feature Request #21776）

- **期望功能**：Claude 在多阶段任务中主动执行检查点 + 上下文重置
- **当前状态**：官方 feature request，尚未实现
- **临时方案**：外部脚本 + /clear 或 /compact

## 八、推荐方案

### 推荐方案：分层组合方案

鉴于当前没有一个方案完美解决全部需求，推荐分层组合：

#### 层 1：执行引擎——Bash + claude -p + Agent SDK

**理由**：这是最成熟、最可控的基础。每一步通过 `claude -p` 独立调用，天然实现独立上下文。

```
核心循环：
  for step in [design, plan, run-plan, check, qa]:
    claude -p "基于 {prev_output} 执行 {step}" \
      --output-format json \
      --allowedTools "适合此步骤的工具列表" \
      > outputs/{step}.json
```

#### 层 2：上下文传递——Handoff 文档模式

**理由**：借鉴 Continuous-Claude 的 handoff 思路，但简化实现。

- 每步结束生成结构化 handoff 文档（Markdown/YAML）
- 下一步启动时将 handoff 文档作为 prompt 前缀注入
- 所有 handoff 文档保存在 `docs/pipeline/` 目录，实现留痕

#### 层 3：质量门控——Custom Subagents

**理由**：利用 Claude Code 原生 subagent 能力定义每个步骤的专用 agent。

- `.claude/agents/designer.md`——设计 agent（只有 Read/Grep/Glob/WebSearch）
- `.claude/agents/planner.md`——计划 agent（只有 Read/Write）
- `.claude/agents/implementer.md`——实现 agent（全部工具）
- `.claude/agents/reviewer.md`——检查 agent（只有 Read/Grep/Bash[test]）
- `.claude/agents/qa-checker.md`——QA agent（只有 Read/Bash）

#### 层 4：循环控制——Hooks + 脚本

**理由**：qa -> fix 循环通过外部脚本 + PostToolUse hooks 实现。

- qa 步骤输出 pass/fail 标志
- 外部脚本根据标志决定是否进入 fix 循环
- fix 循环带 max_retries 限制

#### 层 5（可选增强）：Agent Teams

**理由**：当需要步骤间实时协作时启用，但作为增强而非核心。

---

### 具体实施路径

| 阶段 | 实现内容 | 依赖 | 复杂度 |
|------|---------|------|-------|
| Phase 1 | Bash 编排脚本 + handoff 文档模板 | claude -p | 低 |
| Phase 2 | 每步专用 subagent 定义 | .claude/agents/ | 低 |
| Phase 3 | Hooks 自动触发（PostToolUse auto-format/test） | settings.json | 中 |
| Phase 4 | QA 循环 + fix 循环控制 | Phase 1 脚本 | 中 |
| Phase 5 | Agent Teams 增强（需要并行时） | 实验性功能 | 高 |

---

### 关键设计决策

1. **选 Bash + claude -p 而非 Agent SDK**：初期用 Bash 脚本最快交付，后续可迁移到 Agent SDK
2. **选 handoff 文档而非 session resume**：避免 Issue #21776 描述的上下文膨胀问题
3. **选原生 subagent 而非第三方框架**：减少依赖，利用官方维护的能力
4. **不选 claude-flow/wshobson/agents**：过重，引入成本高于自建编排脚本
5. **关注 obra/superpowers**：作为方法论参考但不直接采用其编排模式
